Code Samples

This section shows the exact bridge flow from Contiki MQTT messages to browser updates.

1) backend/server.js — MQTT Subscriber + WebSocket Bridge

Backend subscribes to the Contiki topic, normalizes payload fields, then pushes JSON to all connected WebSocket clients. This is the central real-time bridge. 

// 1) Read broker address from env (default: local compose broker)
const MQTT_BROKER = process.env.MQTT_BROKER || "mqtt://mosquitto:1883";

// 2) Subscribe to Contiki event topic pattern
mqttClient.subscribe("iot-2/evt/+/fmt/json");

// 3) On every MQTT message, broadcast normalized payload to all WS clients
wss.clients.forEach((client) => {
  if (client.readyState === 1) {
    client.send(JSON.stringify(payload));
  }
});

2) dashboard/index.html — WebSocket Client

Frontend builds the WebSocket URL from the current hostname, so the same page works on localhost and VPS. Incoming data is parsed and inserted into the live table. 

function getWebSocketUrl() {
  // Auto-switch ws / wss and use current host
  var scheme = window.location.protocol === 'https:' ? 'wss' : 'ws';
  return scheme + '://' + window.location.hostname + ':3000';
}

// Open persistent real-time channel
ws = new WebSocket(getWebSocketUrl());

ws.onmessage = function(event) {
  // Parse backend payload and update UI rows
  var data = JSON.parse(event.data);
  updateTable(data);
};

3) contiki_client/run-contiki.sh — Native Build + Run

Contiki native example is patched and built at startup, then launched with TUN prefix/device. This enables MQTT traffic from a native Contiki node inside Docker. 

# Build Contiki native MQTT client
CCACHE_DISABLE=1 make -C "${EXAMPLE_DIR}" TARGET=native clean all

# Start native node with TUN device
exec "${EXAMPLE_DIR}/build/native/mqtt-client.native" --prefix "${TUN_PREFIX}" -t "${TUN_DEV}"
