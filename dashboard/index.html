The Problem: Why Do IoT Devices Need Real-Time Communication?

IoT devices (sensors, actuators, embedded nodes) generate data continuously. A temperature sensor reads every 2 seconds. A heart-rate monitor reads every 100ms. The data is only valuable if it arrives immediately.

Traditional HTTP Approach (Polling):

  Browser                          Server
     |                                |
     |--- GET /data ----------------->|     (request #1)
     |<-- 200 OK {"temp": 24.3} -----|
     |                                |
     |   ... wait 2 seconds ...       |     (wasted time)
     |                                |
     |--- GET /data ----------------->|     (request #2)
     |<-- 200 OK {"temp": 24.5} -----|
     |                                |
     |   ... wait 2 seconds ...       |     (wasted time, wasted bandwidth)
     |                                |
     |--- GET /data ----------------->|     (request #3 -- even if nothing changed!)
     |<-- 200 OK {"temp": 24.5} -----|     (same data, wasted request)

The Solution: Persistent Connections

  • WebSocket — persistent full-duplex connection between browser and server
  • MQTT — lightweight publish-subscribe messaging for IoT devices

Both protocols solve the same core problem: push data instantly when it's available, without the client having to ask repeatedly. They operate at the Application Layer of the OSI model, on top of TCP.

WebSocket: How It Works

WebSocket (RFC 6455) provides full-duplex communication over a single TCP connection. It starts as a regular HTTP request, then upgrades to a persistent connection.

The WebSocket Handshake:

  Browser                                     Server
     |                                           |
     |--- HTTP GET / --------------------------->|
     |   Headers:                                |
     |     Upgrade: websocket                    |
     |     Connection: Upgrade                   |
     |     Sec-WebSocket-Key: dGhlIHN...         |
     |                                           |
     |<-- HTTP 101 Switching Protocols ----------|
     |   Headers:                                |
     |     Upgrade: websocket                    |
     |     Connection: Upgrade                   |
     |     Sec-WebSocket-Accept: s3pPL...        |
     |                                           |
     |========= CONNECTION UPGRADED =============|
     |                                           |
     |   (now full-duplex, both sides can        |
     |    send data at any time)                 |
     |                                           |
     |<-- server pushes: {"temp": 24.3} --------|  (instant, no request needed)
     |<-- server pushes: {"temp": 24.5} --------|  (instant, no request needed)
     |<-- server pushes: {"temp": 25.1} --------|  (instant, no request needed)

Key characteristics:

  • Starts as HTTP, upgrades to WebSocket (port 80 or 443)
  • Full-duplex: both client and server can send messages at any time
  • Persistent: the connection stays open until explicitly closed
  • Low overhead: no HTTP headers on every message (just 2-6 byte frame header)
  • Ideal for: dashboards, chat, gaming, live feeds, IoT monitoring

MQTT: Publish-Subscribe Model

MQTT (Message Queuing Telemetry Transport) is a lightweight messaging protocol designed for constrained devices and low-bandwidth networks. It uses a publish-subscribe pattern — fundamentally different from HTTP's request-response.

How Publish-Subscribe Works:

                         +-------------------+
                         |   MQTT BROKER     |
   PUBLISHERS            |  (Mosquitto)      |            SUBSCRIBERS
                         |                   |
  [Sensor A] --publish-->| topic: room/temp  |--deliver--> [Dashboard]
                         |                   |
  [Sensor B] --publish-->| topic: room/humid |--deliver--> [Logger]
                         |                   |
                         | topic: outdoor/*  |--deliver--> [Mobile App]
                         +-------------------+

  * Publishers don't know who subscribes
  * Subscribers don't know who publishes
  * The Broker handles all routing
  * This DECOUPLING is what makes MQTT ideal for IoT

Why MQTT for IoT?

  • Minimal overhead: 2-byte fixed header (vs HTTP's ~700 bytes of headers)
  • Runs on TCP: reliable delivery on port 1883 (or 8883 with TLS)
  • QoS levels: 0 = at most once, 1 = at least once, 2 = exactly once
  • Retained messages: broker stores last message per topic for new subscribers
  • Last Will & Testament: broker publishes a message if client disconnects unexpectedly
  • Designed for constrained devices: works on microcontrollers with limited RAM/CPU

MQTT Components: Broker, Publisher, Subscriber

Component Role In Our Demo
Broker Central message router. Receives all published messages and delivers them to matching subscribers. Manages connections, topics, QoS, and sessions. Eclipse Mosquitto — open-source MQTT broker running on port 1883. Handles topic matching and fan-out delivery.
Publisher Sends messages to a specific topic on the broker. Does not know or care who is subscribed. Fire-and-forget. Contiki-NG native node — publishes JSON temperature/humidity data to iot-2/evt/status/fmt/json every 2 seconds.
Subscriber Registers interest in one or more topics with the broker. Receives messages automatically when they are published to those topics. Node.js backend (server.js) — subscribes to iot-2/evt/+/fmt/json and bridges received data to WebSocket.

What is Mosquitto?

Eclipse Mosquitto is an open-source message broker that implements MQTT protocol versions 5.0, 3.1.1, and 3.1. It is lightweight, suitable for all devices from low-power single board computers to full servers. It is maintained by the Eclipse Foundation. 

Mosquitto Features:
  - Written in C, extremely lightweight (~120KB binary)
  - Supports MQTT v3.1, v3.1.1, and v5.0
  - TLS/SSL encryption support
  - WebSocket support (browsers can connect directly)
  - Bridge mode (connect multiple brokers)
  - Plugin authentication system
  - Runs on Linux, macOS, Windows, Raspberry Pi

MQTT on Contiki-NG

Contiki-NG is an open-source OS for resource-constrained IoT devices (microcontrollers with ~10KB RAM). It includes a native MQTT client that allows sensor nodes to publish data directly to an MQTT broker.

Real-World Architecture: Contiki Node as MQTT Publisher

  +------------------+         +------------------+         +------------------+
  |  Contiki Node    |  MQTT   |  Mosquitto       |  MQTT   |  Application     |
  |  (Sensor)        |-------->|  Broker           |-------->|  Server          |
  |                  |  :1883  |                  |         |                  |
  |  - Temperature   |         |  Routes messages |         |  - Stores data   |
  |  - Humidity      |         |  to subscribers  |         |  - Dashboard     |
  |  - Light level   |         |                  |         |  - Alerts        |
  +------------------+         +------------------+         +------------------+
     ^                                                           |
     |  Contiki MQTT Client                                      |  WebSocket
     |  - mqtt_connect()                                         |  to browser
     |  - mqtt_publish()                                         v
     |  - mqtt_subscribe()                                  [Web Dashboard]
     |
  Runs on: CC2538, CC2650, Zolertia, nRF52840, etc.

Contiki MQTT Client API:

// Contiki-NG MQTT client example (C language)
#include "mqtt.h"

static struct mqtt_connection conn;

// Connect to broker
mqtt_connect(&conn, "192.168.1.100", 1883, 60);  // broker IP, port, keepalive

// Publish sensor reading
char payload[64];
snprintf(payload, sizeof(payload),
    "{\"temperature\": %.1f}", get_temperature());

mqtt_publish(&conn,
    NULL,                          // request state
    "iot-2/evt/status/fmt/json",   // topic
    (uint8_t *)payload,            // payload
    strlen(payload),               // payload length
    MQTT_QOS_LEVEL_1,              // QoS 1: at least once
    MQTT_RETAIN_OFF);              // don't retain

backend/server.js — MQTT Subscriber + WebSocket Server (Bridge)

const mqtt = require("mqtt");
const { WebSocketServer } = require("ws");

// Connect to Mosquitto broker as MQTT subscriber
const mqttClient = mqtt.connect("mqtt://localhost:1883");
mqttClient.subscribe("iot-2/evt/+/fmt/json");

// Create WebSocket server for browser clients
const wss = new WebSocketServer({ port: 3000 });

// Bridge: MQTT message arrives -> broadcast to all WebSocket clients
mqttClient.on("message", (topic, message) => {
    const payload = message.toString();

    wss.clients.forEach((client) => {
        if (client.readyState === client.OPEN) {
            client.send(payload);  // push to browser instantly
        }
    });
});

dashboard/index.html — WebSocket Client (This Page)

// Open persistent WebSocket connection (HTTP -> 101 Upgrade)
const ws = new WebSocket('ws://localhost:3000');

// Connection established — full-duplex channel is now open
ws.onopen = () => {
    console.log('WebSocket connected!');
};

// Server PUSHES data the instant it arrives from MQTT
// No polling, no repeated requests — just real-time push
ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    // data = { temperature: 24.3, humidity: 55.1, timestamp: "...", device_id: "contiki-01" }
    updateTable(data);
};

// Auto-reconnect if connection drops

    setTimeout(connectWebSocket, 3000);
};
